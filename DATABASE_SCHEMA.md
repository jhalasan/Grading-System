# Grading System - Database Schema Documentation

## Overview
This document outlines the complete database schema for the Grading System application built with PocketBase. The system tracks student grades and maintains an audit log of all modifications.

## Collections

### 1. **users** Collection
User management and role-based access control.

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| id | id | Yes | Auto-generated by PocketBase |
| name | text | Yes | Full name of the user |
| email | email | Yes | Email address (unique) |
| role | select | Yes | Options: `teacher`, `admin` |
| created | date | Auto | Auto-generated timestamp |
| updated | date | Auto | Auto-updated timestamp |

**Indexes**: email (unique)

### 2. **students** Collection
Student information and course enrollment.

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| id | id | Yes | Auto-generated by PocketBase |
| student_name | text | Yes | Full name of student |
| course | text | Yes | Course name (e.g., "Math 101", "Physics 201") |
| section | text | Yes | Section identifier (e.g., "A", "B", "01", "02") |
| created | date | Auto | Auto-generated timestamp |
| updated | date | Auto | Auto-updated timestamp |

**Indexes**: course, section

### 3. **grades** Collection
Student grades and academic records.

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| id | id | Yes | Auto-generated by PocketBase |
| student_id | relation | Yes | Reference to `students` collection |
| subject | text | Yes | Subject/Course name (e.g., "Exam 1", "Assignment 2") |
| grade_value | number | Yes | Grade score (0-100) |
| last_modified_by | relation | Yes | Reference to `users` collection (who last edited) |
| created | date | Auto | Auto-generated timestamp |
| updated_at | date | Auto | Auto-updated timestamp |

**Relations**:
- `student_id` → `students.id` (many-to-one)
- `last_modified_by` → `users.id` (many-to-one)

**Indexes**: student_id, last_modified_by, updated_at

### 4. **activity_logs** Collection
Audit trail for all grade modifications. Used for security monitoring and detecting unauthorized changes.

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| id | id | Yes | Auto-generated by PocketBase |
| user_id | relation | Yes | Reference to `users` collection |
| action_type | select | Yes | Options: `CREATE_GRADE`, `UPDATE_GRADE`, `DELETE_GRADE`, `LOGIN`, `LOGOUT` |
| record_id | text | Yes | ID of the grade being modified |
| old_value | text | Yes | Previous grade value (JSON string) |
| new_value | text | Yes | New grade value (JSON string) |
| timestamp | date | Yes | Exact time of action |
| ip_address | text | No | IP address of the request (for security auditing) |
| device | text | No | Device information (Device Type - OS - Browser) |
| created | date | Auto | Auto-generated timestamp |

**Relations**:
- `user_id` → `users.id` (many-to-one)

**Indexes**: user_id, record_id, action_type, timestamp

## Data Flow - Grade Update Example

1. **Teacher edits a grade** in the `grades` collection:
   - Grade value: 85 → 92
   - Updated `last_modified_by` to teacher's user ID
   - Updated `updated_at` timestamp

2. **System automatically creates audit log entry**:
   ```json
   {
     "user_id": "teacher_id_123",
     "action_type": "UPDATE_GRADE",
     "record_id": "grade_id_456",
     "old_value": "{\"grade_value\": 85}",
     "new_value": "{\"grade_value\": 92}",
     "timestamp": "2026-04-09T14:30:00Z",
     "ip_address": "192.168.1.100"
   }
   ```

3. **Query with expand relations** retrieves enriched data:
   - Grade record with expanded `student_id` shows student details
   - Grade record with expanded `last_modified_by` shows teacher name
   - Activity log with expanded `user_id` shows who made the change

## Security Considerations

1. **Immutable Logs**: Activity logs should never be edited or deleted (PocketBase rules)
2. **IP Tracking**: Helps identify unauthorized access attempts
3. **Timestamp Fields**: Both `created` and `timestamp` are used for verification
4. **User Association**: All changes traced to specific user accounts
5. **Role-Based Rules**: Only teachers and admins can modify grades

## PocketBase Setup Instructions

### Collection Rules (Example)

**grades** - Update Rule:
```
@request.auth.id != null && 
(@request.auth.role = "teacher" || @request.auth.role = "admin")
```

**grades** - Delete Rule:
```
@request.auth.id != null && @request.auth.role = "admin"
```

**activity_logs** - Create Rule:
```
@request.auth.id != null
```

**activity_logs** - Delete Rule:
```
false
```
(Prevent deletion of audit logs)

## API Queries

### Fetch all grades with student and user details
```
GET /api/collections/grades/records
?expand=student_id,last_modified_by
&sort=-updated_at
```

### Get activity logs for a specific grade
```
GET /api/collections/activity_logs/records
?filter=record_id="grade_id_123"
&expand=user_id
&sort=-timestamp
```

### Get all changes made by a teacher
```
GET /api/collections/activity_logs/records
?filter=user_id="user_id_456"
&expand=user_id
&sort=-timestamp
```

### Get all grade modifications in a date range
```
GET /api/collections/activity_logs/records
?filter=timestamp>="2026-04-01"&&timestamp<"2026-04-30"
&expand=user_id
&sort=-timestamp
```

## Notes for Disaster Simulation

- **Scenario**: Someone unknowingly modifies a student's grade
- **Detection**: Instantly visible in Activity Logs
- **Recovery**: Old values stored in `old_value` field for verification
- **Accountability**: `user_id` shows exactly who made the change
- **Traceability**: Timestamp and IP address provide additional verification
